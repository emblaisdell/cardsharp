# @cardsharp/ml — classical-ML players

Lightweight, dependency-free strategy learning for ♠# games. No neural nets:
a **linear softmax policy** over hand-crafted features, trained by **self-play
REINFORCE** (policy gradient). A trained model is a JSON weight vector of a few
dozen floats (≈1 KB) — small enough to ship to the browser and run with a dot
product.

## Why this works for any ♠# game

The engine already turns every game into a sequential decision process: at each
decision point it hands a controller `(observation, legalMoves)` and resumes with
the choice; at the end each seat gets a win/loss reward. So learning a player is
just:

1. **Featurize** each legal option into a fixed-length vector φ(obs, option)
   ([features.ts](src/features.ts)) — generic features read from the observation
   (option's rank/suit/value, a target player's public card count, my hand
   size/value, turn, number of options…), so one architecture fits all games.
2. **Score & pick**: `score(option) = w · φ`; softmax to explore while training,
   argmax to play ([policy.ts](src/policy.ts)).
3. **Learn**: play self-play games, then nudge weights toward choices that led to
   wins — `w += α · (reward − baseline) · (φ_chosen − Σ_i p_i φ_i)`
   ([trainer.ts](src/trainer.ts)).

The same `MLController` is just another `Controller`, so it drops into the CLI,
the browser, or a mixed table against humans/bots.

## A second player: Monte-Carlo Tree Search

[mcts.ts](src/mcts.ts) is a **general UCT search** that works on any ♠# game with
no game-specific code. The engine is deterministic given (program, seed,
answer-sequence), so at a decision the controller **re-simulates** the game —
replaying the recorded answer history into a fresh game to reproduce the current
situation — tries a candidate action, and plays out. Growing a UCT tree over this
player's future decisions (opponents + chance sampled) gives an action value.

It trains nothing; it spends compute **per move** instead of up front. It is a
**perfect-information ("cheating") baseline**: re-simulation uses the real seed,
so each playout sees the true shuffle and opponents' hidden cards — a strong
*upper bound*, not a fair imperfect-information agent. That makes it the ideal
reference for "how much skill does this game actually reward?" when comparing
learners.

### Fairness boundary

The bots you actually play (`RandomController`, `MLController`) are **fair**: their
only input is `(req, obs)`, and `obs` is the visibility-masked observation —
opponents' hands and face-down piles come through as `null`. A regression test
([packages/core/test/fairness.test.ts](../core/test/fairness.test.ts)) asserts no
hidden card ever leaks into an observation.

`MctsController` is the exception and is **not fair** — it re-simulates the true
world. To prevent it being wired in as a "fair" bot by accident, its constructor
**throws unless you pass `perfectInfo: true`**. A *fair* information-set MCTS —
one that searches a determinization built from the player's observation, with a
fresh local RNG and never the seed — requires a **resumable engine** (so search
can continue from a reconstructed mid-game state). That's planned; until then,
MCTS is benchmark-only.

## Usage

```bash
# train and save a model, then report win-rate vs random opponents
node packages/ml/src/main.ts train games/blackjack.card --games 15000 --out models/blackjack.json

# evaluate a saved model
node packages/ml/src/main.ts eval  games/thirtyone.card --model models/thirtyone.json --games 2000

# head-to-head: random vs linear vs MCTS, with compute reported
node packages/ml/src/main.ts compare games/blackjack.card --model models/blackjack.json --sims 80 --games 200

# train ALL games (longer) + benchmark + write docs/ml-benchmarks.md
node packages/ml/bench.mjs
```

## Results & compute

Full, reproducible numbers — win-rate of each method vs random opponents **and
the exact training compute** (games, decisions, wall-time) — live in
**[docs/ml-benchmarks.md](../../docs/ml-benchmarks.md)**, regenerated by
`node packages/ml/bench.mjs`. Headline shape: Linear PG adds a few points in
low-signal games and tens of points where the features capture the key decision
(Thirty-One); perfect-information MCTS shows the headroom above that.

Lift varies by how much skill the game rewards and how well the generic features
capture it. Games dominated by hidden information or by decisions the linear
model can't see (multi-card selects, numeric ranges — handled by a random
fallback) show smaller gains; per-game feature additions in
[features.ts](src/features.ts) are the obvious next lever. MCTS is omitted for the
long multi-round games (Crazy Bridge, Money Money Money) because one search move
re-simulates a whole game — its cost scales with game length.

## Browser

`LinearPolicy.fromJSON(model)` + `featurizeOptions(req, obs)` are plain
TypeScript with no Node dependencies, so the web client loads a model JSON and
runs the policy inline — see `packages/web`.
